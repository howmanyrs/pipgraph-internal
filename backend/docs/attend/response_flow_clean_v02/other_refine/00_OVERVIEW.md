## Краткое описание

Система многоуровневых подтверждений (User Check System) для PipGraph — это архитектура взаимодействия с пользователем при обработке заметок из Obsidian. Система обеспечивает:

1.  **Классификацию заметок** по методу PARA (Project/Area/Resource/Archive).
2.  **Извлечение сущностей** из текста заметки (Person, Organization, Task и др.).
3.  **Многоуровневое уточнение** информации с сохранением полной истории действий пользователя.
4.  **Асинхронную обработку** с возможностью прерывания и возобновления процесса.

### Ключевые принципы MVP

-   ✅ **Human-in-the-loop**: Пользователь подтверждает и корректирует действия LLM.
-   ✅ **Графовая архитектура**: Все метаданные хранятся как отдельные ноды в Neo4j.
-   ✅ **Полная история**: Каждое действие пользователя сохраняется для аудита и обучения.
-   ✅ **Прерываемый workflow**: LangGraph с `interrupt`/`resume` для долгих операций.
-   ✅ **Приоритизация**: Важные вопросы задаются первыми, несущественные пропускаются.

## Ключевые термины

### Уровни подтверждений

-   **Level 1 (L1): PARA Classification**
    Определение типа заметки (Project/Area/Resource/Archive).

-   **Level 2 (L2): Container Assignment**
    Привязка заметки к конкретному проекту, области или ресурсу.

-   **Level 3 (L3): Entity Confirmation**
    Подтверждение извлеченных сущностей (Person, Organization, Task и др.).

-   **Level 4 (L4): Attribute Validation** _(Не в MVP)_
    Проверка конкретных атрибутов сущностей (email, role и т.д.).

### Статусы подтверждений

-   **pending**: Ожидает подтверждения пользователя.
-   **confirmed**: Подтверждено пользователем без изменений.
-   **modified**: Подтверждено с изменениями.
-   **rejected**: Отклонено пользователем.
-   **skipped**: Пропущено (отложено на потом).
-   **auto_confirmed**: Автоматически подтверждено (высокая уверенность).

### Основные сущности

-   **UserCheckStatus**: Нода в Neo4j, представляющая одно событие подтверждения.
-   **PARA Container**: `Project`, `Area`, `Resource` — высокоуровневые узлы-агрегаторы.
-   **EntityNode**: Извлеченная из текста сущность (Person, Organization, Task и др.).
-   **FieldModification**: Структура, описывающая изменение одного поля сущности.

## Что НЕ входит в MVP

Следующие функции запланированы для пост-MVP:

-   ❌ **Level 4 (Attribute Validation)**: Детальная проверка атрибутов.
-   ❌ **Batch clarifications**: Показ нескольких вопросов одновременно.
-   ❌ **Миграции**: Автоматический перенос старых данных.
-   ❌ **Performance optimizations**: Расширенная оптимизация запросов (кроме базовых индексов).
-   ❌ **ML-based learning**: Обучение на основе исправлений пользователя.
-   ❌ **Advanced analytics**: Подробная аналитика и метрики.
-   ❌ **Reclassification UI**: Интерфейс для изменения классификации старых заметок.

MVP фокусируется на **работающем ядре** системы, которое решает основные задачи:
1.  Классификация заметок (L1).
2.  Привязка к проектам (L2).
3.  Подтверждение сущностей (L3).
4.  Сохранение истории действий.
5.  Асинхронная обработка с прерыванием.

## Критерии успеха MVP

Рабочая демонстрация, где:

1.  ✅ Заметка отправлена на сервер через WebSocket.
2.  ✅ Сущности извлечены из текста (LLM + Graphiti).
3.  ✅ Система запрашивает уточнения последовательно (L1 → L2 → L3).
4.  ✅ Пользователь отключается от сервера.
5.  ✅ **Через 1 день** пользователь подключается снова.
6.  ✅ Система показывает незавершенные вопросы для этой заметки.
7.  ✅ Пользователь отвечает на вопросы.
8.  ✅ Обработка завершается, данные сохранены в Neo4j.
9.  ✅ Вся история действий пользователя доступна для аудита.

**Технические требования:**
-   Состояние LangGraph сохраняется в `AsyncSqliteSaver`.
-   `interrupt`/`resume` работает корректно.
-   `UserCheckStatus` ноды создаются для каждого действия.
-   История связана через отношения `:NEXT`.
-   Текущий статус помечен `is_current: true`.
-   Запросы к Neo4j выполняются за разумное время (< 200ms для dashboard).
