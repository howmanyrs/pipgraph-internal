# Фазы реализации MVP

**Дата создания:** 2025-11-17
**Статус:** План для реализации
**Версия:** 1.0

---

## Введение

Этот документ описывает **поэтапный план реализации MVP** системы многоуровневых подтверждений. План разбит на четыре фазы, каждая из которых добавляет инкрементальную ценность и завершается работающей функциональностью.

**Общая длительность MVP (фазы 1-3):** 5-7 недель.

---

## Phase 1: Базовая инфраструктура и Workflow

**Длительность:** 2-3 недели
**Цель:** Создать работающий "скелет" системы: хранение статусов и асинхронный workflow с `interrupt`/`resume`.

### Ключевые задачи

1.  **Модели данных:**
    -   Реализовать Pydantic-модели: `UserCheckStatus`, `FieldModification`.

2.  **Neo4j Схема:**
    -   Создать миграцию для ноды `:UserCheckStatus`.
    -   Настроить базовые `constraints` (уникальность `id`) и `indexes` (на `status`, `timestamp`).

3.  **CRUD операции:**
    -   Реализовать функции для создания и обновления `UserCheckStatus` нод, включая логику `is_current` и цепочку `[:NEXT]`.

4.  **LangGraph Workflow (базовый):**
    -   Создать `NoteProcessingState` (TypedDict).
    -   Реализовать базовые узлы: `extract` (заглушка), `check_clarification` (заглушка), `request_clarification` (с `interrupt`), `process_response`, `finalize`.
    -   Собрать граф и подключить `AsyncSqliteSaver`.

5.  **WebSocket API:**
    -   Создать эндпоинт для приема запросов на обработку и ответов на уточнения.

### Критерии готовности

-   ✅ `UserCheckStatus` ноды и связи `[:HAS_CHECK]`, `[:NEXT]` корректно создаются в Neo4j.
-   ✅ LangGraph workflow запускается, останавливается на `interrupt` и успешно возобновляется после получения ответа.
-   ✅ Состояние workflow персистентно сохраняется в SQLite.
-   ✅ **End-to-end тест:** Отправка заметки → получение mock-вопроса → отправка ответа → workflow завершается.

### Что НЕ входит в эту фазу

-   ❌ Реальная логика L1/L2/L3. Используются заглушки.
-   ❌ Приоритизация и auto-confirm.

---

## Phase 2: PARA-классификация (L1 и L2)

**Длительность:** 2 недели
**Цель:** Реализовать классификацию заметок по методу PARA и привязку к контейнерам.

### Ключевые задачи

1.  **Модели данных:**
    -   Реализовать Pydantic-модели: `Project`, `Area`, `Resource`.

2.  **Neo4j Схема:**
    -   Добавить в миграцию ноды `:Project`, `:Area`, `:Resource` и связь `[:IS_PART_OF]`.
    -   Настроить `constraints` и `indexes` для PARA-контейнеров.

3.  **CRUD операции:**
    -   Реализовать функции для создания/поиска PARA-контейнеров и связи заметок с ними.

4.  **Сервисы:**
    -   Создать сервис для L1-классификации (можно начать с простых эвристик на ключевых словах).
    -   Создать сервис для поиска существующих контейнеров для L2.

5.  **LangGraph Workflow (расширение):**
    -   Добавить реальную логику в `extract_entities_node` для вызова L1/L2 сервисов.
    -   Реализовать логику в `check_clarification_node` для формирования вопросов L1 и L2.
    -   Реализовать логику в `process_response_node` для обработки ответов L1 и L2 (создание/привязка контейнеров).

### Критерии готовности

-   ✅ Ноды `Project`, `Area`, `Resource` и связи `[:IS_PART_OF]` создаются в Neo4j.
-   ✅ Система корректно задает вопросы L1 (тип заметки) и L2 (выбор контейнера).
-   ✅ Ответы пользователя приводят к правильной привязке заметки к контейнеру.
-   ✅ **End-to-end тест:** Отправка заметки → ответ на L1-вопрос → ответ на L2-вопрос → заметка успешно привязана к проекту в Neo4j.

### Что НЕ входит в эту фазу

-   ❌ Подтверждение сущностей (L3).

---

## Phase 3: Подтверждение сущностей (L3) и Приоритизация

**Длительность:** 1-2 недели
**Цель:** Добавить основной `human-in-the-loop` функционал — подтверждение извлеченных сущностей с умной приоритизацией.

### Ключевые задачи

1.  **Сервисы:**
    -   Реализовать логику приоритизации вопросов (`ENTITY_PRIORITY`).
    -   Реализовать логику автоматического подтверждения (`should_auto_confirm`).

2.  **LangGraph Workflow (расширение):**
    -   В `extract_entities_node` применить `auto-confirm` и создать `UserCheckStatus` для остальных сущностей.
    -   В `check_clarification_node` добавить логику формирования L3-вопросов и их сортировки согласно приоритетам.
    -   В `process_response_node` реализовать обработку ответов L3 (`confirm`, `modify`, `reject`, `skip`).

3.  **CRUD операции:**
    -   Расширить CRUD для `UserCheckStatus`, чтобы корректно обрабатывать `modifications` при изменении сущности.

### Критерии готовности

-   ✅ L3-уточнения генерируются для всех `pending` сущностей.
-   ✅ Вопросы задаются в правильном порядке: L1 → L2 → L3 (с приоритизацией внутри).
-   ✅ Auto-confirm работает для высокоуверенных и низкоприоритетных сущностей.
-   ✅ Пользователь может подтверждать, изменять и отклонять сущности.
-   ✅ **End-to-end тест:** Отправка заметки с несколькими сущностями → прохождение L1/L2 → ответы на L3-вопросы → все сущности в Neo4j имеют корректный финальный статус.

## Phase 4 (Опционально / Пост-MVP): Улучшения UX и Аналитика

**Длительность:** 2 недели
**Цель:** Повысить удобство использования системы и добавить базовые инструменты для анализа.

### Ключевые задачи

1.  **Механизм Skip/Defer:**
    -   Добавить в `UserCheckStatus` поля `skip_count`, `defer_until`.
    -   Реализовать логику обработки `skip` и `defer` действий в `process_response_node`.

2.  **Batch Clarifications (опционально):**
    -   Реализовать логику группировки однотипных L3-вопросов (например, подтвердить всех `Person` на одном экране).

3.  **Аналитика:**
    -   Создать API-эндпоинты для получения базовой статистики:
        -   Количество статусов по типам (`pending`, `confirmed` и т.д.).
        -   Самые часто изменяемые поля.
        -   Средняя уверенность системы по типам сущностей.

### Критерии готовности

-   ✅ Пользователь может пропускать вопросы или откладывать их.
-   ✅ Базовые метрики производительности системы доступны через API.
