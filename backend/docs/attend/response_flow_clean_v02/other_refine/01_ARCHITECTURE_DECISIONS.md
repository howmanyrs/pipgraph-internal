## Решение 1: UserCheck Status как отдельные ноды

### Суть решения

**Вариант 2: "Нода Статуса"** — каждое событие подтверждения пользователя хранится как **отдельная нода** `(:UserCheckStatus)` в Neo4j.

### Статус

✅ **ОДОБРЕНО**

Цитата пользователя из `response_to_user_check_granularity_proposal.md`:
> "Спасибо, Вариант 2: 'Нода Статуса' - это то что нужно в данный момент!"

### Структура

```cypher
// Сущность и текущий статус
(entity:EntityNode)-[:HAS_CHECK {is_current: true}]->(current:UserCheckStatus)

// История изменений (от нового к старому)
(current)-[:NEXT]->(previous:UserCheckStatus)-[:NEXT]->(older:UserCheckStatus)
```

**Ключевые элементы:**
-   Свойство `is_current: true` на связи `[:HAS_CHECK]` указывает на актуальный статус для быстрых запросов.
-   Цепочка связей `[:NEXT]` сохраняет полную, неизменяемую историю действий пользователя.
-   Каждая нода `UserCheckStatus` содержит полный snapshot состояния на момент действия.

### Почему это решение

#### ✅ Преимущества

1.  **Производительность запросов**: Все поля индексируются напрямую. Запросы типа "найти все pending статусы" выполняются мгновенно, без сканирования JSON.
2.  **Полная история (Audit Trail)**: Каждое действие (`pending` → `skipped` → `confirmed`) сохранено как отдельная нода. Это критично для аналитики, отладки и будущего обучения ML-моделей.
3.  **Гибкость аналитики**: Легко выполнять агрегирующие запросы ("какие поля чаще всего корректируются?", "какие типы сущностей чаще пропускают?").
4.  **Соответствие графовой модели**: Метаданные представлены как первоклассные граждане графа (ноды), что является идиоматичным для Neo4j.

#### ❌ Отклоненные альтернативы

-   **Хранение в атрибутах (JSON)**: Невозможно индексировать вложенные поля, нет истории.
-   **Типизированные отношения (`[:CONFIRMED]`)**: Теряется история при смене статуса.

## Решение 2: PARA entities как отдельные ноды

### Суть решения

Высокоуровневые контейнеры PARA (`Project`, `Area`, `Resource`) хранятся как **отдельные ноды** в Neo4j, а не как атрибуты заметки. Заметки связываются с ними через отношение `[:IS_PART_OF]`.

### Статус

✅ **ОДОБРЕНО**

Это решение соответствует архитектуре, описанной в `PARA_TYPES_ARCHITECTURE.md`.

### Структура

```cypher
// PARA контейнер как отдельная нода
(p:Project {id: "proj_123", name: "Q4 Marketing Campaign"})

// Заметка ссылается на контейнер
(n:Note {path: "meetings/sync.md"})

// Основная связь - источник истины
(n)-[:IS_PART_OF]->(p)
```

**Гибридный подход:** Для ускорения фильтрации на ноде `Note` могут кэшироваться `para_type` и `project_id`, но **источником истины** всегда является отдельная нода контейнера.

### Почему это решение

#### ✅ Преимущества

1.  **Целостность данных (Single Source of Truth)**: Один проект = один узел. Изменение дедлайна проекта обновляет только одну ноду, и все связанные заметки автоматически видят это изменение.
2.  **Мощные графовые запросы**: Позволяет выполнять сложные запросы, например, "найти все задачи во всех заметках проекта X, назначенные на Y".
3.  **Масштабируемость**: Легко добавлять новые атрибуты к проектам (бюджет, команда) или связи между ними (`(p1)-[:DEPENDS_ON]->(p2)`), не затрагивая схему заметок.

#### ❌ Отклоненные альтернативы

-   **Хранение в атрибутах заметки**: Приводит к дублированию данных, отсутствию единого источника истины и усложнению обновлений.

### Источники

-   `user_check_granularity_proposal.md` (раздел 4)
-   `PARA_TYPES_ARCHITECTURE.md`

## Решение 3: Многоуровневая система подтверждений (L1-L3)

### Суть решения

Система уточнений организована в **три последовательных уровня гранулярности**, которые реализуются в MVP. Уровень L4 откладывается на пост-MVP.

**Последовательность:** L1 → L2 → L3

### Статус

✅ **ОДОБРЕНО** (L1-L3 для MVP)
⏸️ **ОТЛОЖЕНО** (L4 для пост-MVP)

### Уровни подтверждений

-   **Level 1: PARA Classification**
    **Задача:** Определить тип заметки (`Project`, `Area`, `Resource`).
    **Вопрос пользователю:** `Эта заметка - Проект, Область или Ресурс?`

-   **Level 2: Container Assignment**
    **Задача:** Привязать заметку к конкретному контейнеру.
    **Вопрос пользователю:** `К какому проекту отнести заметку? Создать новый или выбрать из существующих?`

-   **Level 3: Entity Confirmation**
    **Задача:** Подтвердить извлеченные из текста сущности (`Person`, `Organization`).
    **Вопрос пользователю:** `Подтвердите сущность "Person: John Smith"?`

-   **Level 4: Attribute Validation** _(Не в MVP)_
    **Задача:** Проверить конкретные атрибуты сущности (email, role). Отложено, так как MVP может обойтись редактированием на L3.

### Почему это решение

#### ✅ Преимущества

1.  **Структурированность**: Четкая иерархия вопросов от общего к частному (сначала структура, потом детали).
2.  **Управляемый User Flow**: Пользователь не перегружен информацией, так как решает одну задачу за раз.
3.  **Приоритизация**: Гарантирует, что наиболее важные структурные вопросы (L1, L2) задаются первыми.

### Источники

-   `user_check_granularity_proposal.md` (раздел 2)
-   `per_node_confirmation_overview.md`

## Решение 4: Приоритизация уточнений и Auto-Confirm

### Суть решения

Вопросы пользователю **приоритизируются** на основе типа сущности и уверенности системы. Несущественные и высокоуверенные сущности подтверждаются **автоматически**.

### Статус

✅ **ОДОБРЕНО**

### Логика

1.  **Приоритеты сущностей (L3):**
    -   **Высокий:** `Person`, `Organization`
    -   **Средний:** `Task`, `Decision`
    -   **Низкий:** `Idea`, `Source`, `Question`

2.  **Автоматическое подтверждение (`auto_confirmed`):**
    -   Сущности с **низким приоритетом** и **очень высокой уверенностью** (`> 0.95`) не требуют подтверждения.
    -   **Пример:** `Source("https://example.com")` с уверенностью 0.96 будет подтвержден автоматически. `Person("John Smith")` с уверенностью 0.85 потребует ручного подтверждения.

### Почему это решение

#### ✅ Преимущества

1.  **Экономия времени пользователя**: Фокус на том, что действительно требует человеческого внимания.
2.  **Улучшенный UX**: Снижается "усталость от подтверждений", процесс становится менее монотонным.
3.  **Прозрачность**: Авто-подтверждения помечаются соответствующим статусом и могут быть просмотрены или отменены.

### Источники

## Решение 5: LangGraph с interrupt/resume

### Суть решения

Использовать **LangGraph** для оркестрации workflow обработки заметки. Это позволяет **прерывать** (`interrupt`) выполнение для получения ответа от пользователя и **возобновлять** (`resume`) его позже, сохраняя состояние.

### Статус

✅ **ОДОБРЕНО**

### Технологический стек

-   **LangGraph**: Оркестрация графа состояний.
-   **AsyncSqliteSaver**: Хранение состояния workflow для MVP.
-   **FastAPI WebSocket**: Коммуникация с клиентом в реальном времени.

### Почему это решение

#### ✅ Преимущества

1.  **Асинхронность**: Пользователь может ответить на вопрос через час или день — workflow продолжится с того же места.
2.  **Надежность**: Состояние процесса сохраняется персистентно, что защищает от сбоев или перезапусков сервера.
3.  **Простота реализации**: LangGraph предоставляет встроенные механизмы для `interrupt` и `resume`, избавляя от необходимости писать сложную логику управления состоянием вручную.

### Источники

-   `user_check_mvp_plan.md`
-   `state_serialization_details.md`
