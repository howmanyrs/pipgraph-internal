

### **Архитектура Системы Знаний на основе Графов и Метода PARA**

Этот документ описывает формальную модель данных для создания интеллектуальной системы управления знаниями. Цель архитектуры — преобразовать разрозненные заметки в единый, взаимосвязанный граф знаний, где информация структурирована не по папкам, а по смыслу и контексту.

#### **1. Метод PARA как Архитектурный Принцип**

В нашей архитектуре метод PARA — это не просто система папок, а **набор высокоуровневых узлов-агрегаторов**, которые определяют роль, контекст и жизненный цикл информации в графе. Каждый фрагмент информации (атомарная заметка или блок) получает свой смысл через связь с одним из этих узлов.

*   **`Project` (Проект):** Узел, представляющий собой конкретную цель с четким сроком выполнения. Он объединяет все блоки контента, задачи и решения, необходимые для достижения этой цели. Когда цель достигнута, все связанные с проектом блоки могут быть архивированы.
    *   *Пример:* "Запуск маркетинговой кампании Q4", "Подготовка к конференции TechLead".

*   **`Area` (Область):** Узел, представляющий сферу ответственности без конечной даты. Он агрегирует знания, стандарты и заметки, которые требуют постоянного внимания и развития.
    *   *Пример:* "Здоровье и фитнес", "Управление командой", "Личные финансы".

*   **`Resource` (Ресурс):** Узел, объединяющий информацию по теме, представляющей интерес, но не требующей немедленных действий. Это ваша личная библиотека знаний.
    *   *Пример:* "Квантовые вычисления", "История Древнего Рима", "Рецепты веганской кухни".

*   **`Archive` (Архив):** Это не узел, а **состояние**. Завершенные проекты, неактуальные области и устаревшие ресурсы не удаляются. Вместо этого их связь с активными узлами `Project` или `Area` может быть изменена или помечена как архивная, что убирает их из активного поля зрения, но сохраняет в графе для поиска.

#### **2. Формальная Модель Данных (Схема Графа)**

Модель состоит из типов узлов (Labels) и типов отношений (Edges).

##### **2.1. Типы Узлов (Nodes / Labels)**

**Высокоуровневые (PARA) Узлы-Агрегаторы:**

| Label | Описание | Ключевые Свойства (Properties) |
| :--- | :--- | :--- |
| **`Project`** | Конкретная цель с дедлайном. | `name: string`, `status: string (active, completed, archived)`, `deadline: date` |
| **`Area`** | Сфера ответственности. | `name: string`, `goal: string (описание цели области)` |
| **`Resource`** | Тема или интерес. | `topic: string` |

**Гранулярные Узлы Контента:**

| Label | Описание | Ключевые Свойства (Properties) |
| :--- | :--- | :--- |
| **`SourceFile`** | Представляет исходный файл (например, `note.md`). | `path: string`, `createdAt: datetime`, `updatedAt: datetime` |
| **`ContentBlock`**| **Центральный узел.** Атомарный блок информации (абзац, пункт списка). | `content: text`, `blockId: string` (уникальный ID в файле, `^1a2b3c`) |

**Извлекаемые Сущности (Entities):**

| Label | Описание | Ключевые Свойства (Properties) |
| :--- | :--- | :--- |
| **`Person`** | Человек, контакт. | `name: string`, `role: string`, `contact: string` |
| **`Organization`** | Компания, команда, институт. | `name: string` |
| **`Task`** | Конкретное действие к выполнению. | `description: string`, `status: string (todo, done)`, `dueDate: date` |
| **`Decision`** | Зафиксированное решение. | `statement: string`, `decisionDate: date` |
| **`Source`** | Внешний источник информации. | `type: string (url, book)`, `identifier: string` |
| **`Idea`** | Ключевая мысль, тезис. | `summary: string` |
| **`Question`** | Сформулированный вопрос. | `text: string` |

##### **2.2. Типы Отношений (Edges / Relationships)**

**Структурные Отношения:**

| Структура | Описание |
| :--- | :--- |
| `(ContentBlock) -[:CONTAINED_IN]-> (SourceFile)` | Блок содержится в этом исходном файле. |
| `(ContentBlock) -[:IS_PART_OF]-> (Project \| Area \| Resource)` | Блок является частью этого проекта, области или ресурса. **Это ключевая связь PARA.** |
| `(ContentBlock) -[:RELATES_TO]-> (ContentBlock)` | Два блока семантически связаны (бэклинк). |

**Отношения от Контента к Сущностям:**

| Структура | Описание |
| :--- | :--- |
| `(ContentBlock) -[:MENTIONS]-> (Person \| Organization)` | Блок упоминает человека или организацию. |
| `(ContentBlock) -[:CONTAINS_TASK]-> (Task)` | Блок содержит описание задачи. |
| `(ContentBlock) -[:DOCUMENTS]-> (Decision)` | Блок документирует принятое решение. |
| `(ContentBlock) -[:REFERENCES]-> (Source)` | Блок ссылается на внешний источник. |
| `(ContentBlock) -[:DEFINES]-> (Idea)` | Блок формулирует ключевую идею. |
| `(ContentBlock) -[:ASKS]-> (Question)` | Блок поднимает вопрос. |

**Отношения между Сущностями:**

| Структура | Описание |
| :--- | :--- |
| `(Task) -[:IS_ASSIGNED_TO]-> (Person)` | Задача назначена на этого человека. |
| `(Person) -[:WORKS_AT]-> (Organization)` | Человек работает в этой организации. |

#### **3. Практический Пример: Жизненный Цикл Заметки**

Рассмотрим, как заметка из Obsidian будет представлена в графе.

**Шаг 1: Исходные данные (файл `meetings/Project Alpha Sync.md`)**

```markdown
# Встреча по Проекту Альфа - 20.10.2023

## Участники
- [[Анна Смирнова]]
- [[Иван Петров]] из [[ООО ТехноСфера]]

## Решения
- **Решено:** Переходим на новый API Gateway. ^decision1

## Задачи
- [ ] [[Анна Смирнова]] Подготовить презентацию по новому API к 2024-11-01 ^task1
- [ ] Проанализировать конкурентов. См. статью от [[Gartner]] (https://gartner.com/...) ^task2

## Мысли
- Ключевая идея: **простота интеграции для внешних команд** важнее производительности. ^idea1
```

**Шаг 2: Декомпозиция и Создание Узлов**

Система анализирует файл и создает следующие узлы:

*   **Высокоуровневые узлы:**
    *   `Project(name: "Проект Альфа", status: "active")`
*   **Узлы контента:**
    *   `SourceFile(path: "meetings/Project Alpha Sync.md")`
    *   `ContentBlock(content: "Переходим на новый API Gateway.", blockId: "^decision1")`
    *   `ContentBlock(content: "Подготовить презентацию...", blockId: "^task1")`
    *   `ContentBlock(content: "Проанализировать конкурентов...", blockId: "^task2")`
    *   `ContentBlock(content: "простота интеграции... важнее...", blockId: "^idea1")`
*   **Извлеченные сущности:**
    *   `Person(name: "Анна Смирнова")`
    *   `Person(name: "Иван Петров")`
    *   `Organization(name: "ООО ТехноСфера")`
    *   `Decision(statement: "Переходим на новый API Gateway", decisionDate: "2023-10-20")`
    *   `Task(description: "Подготовить презентацию...", dueDate: "2024-11-01")`
    *   `Task(description: "Проанализировать конкурентов...")`
    *   `Source(type: "url", identifier: "https://gartner.com/...")`
    *   `Organization(name: "Gartner")`
    *   `Idea(summary: "Простота интеграции для внешних команд важнее производительности")`

**Шаг 3: Создание Отношений**

Затем система строит связи между этими узлами:

1.  **Структурные связи:**
    *   `(Все ContentBlock'и) -[:CONTAINED_IN]-> (SourceFile: "meetings/...")`
    *   `(Все ContentBlock'и) -[:IS_PART_OF]-> (Project: "Проект Альфа")`
2.  **Связи контента и сущностей:**
    *   `(ContentBlock: ^decision1) -[:DOCUMENTS]-> (Decision: "Переходим...")`
    *   `(ContentBlock: ^task1) -[:CONTAINS_TASK]-> (Task: "Подготовить...")`
    *   `(ContentBlock: ^task2) -[:CONTAINS_TASK]-> (Task: "Проанализировать...")`
    *   `(ContentBlock: ^task2) -[:MENTIONS]-> (Organization: "Gartner")`
    *   `(ContentBlock: ^task2) -[:REFERENCES]-> (Source: "gartner.com/...")`
    *   `(ContentBlock: ^idea1) -[:DEFINES]-> (Idea: "Простота...")`
3.  **Связи между сущностями:**
    *   `(Task: "Подготовить...") -[:IS_ASSIGNED_TO]-> (Person: "Анна Смирнова")`
    *   `(Person: "Иван Петров") -[:WORKS_AT]-> (Organization: "ООО ТехноСфера")`

В результате мы получаем богатый, взаимосвязанный граф, где каждый элемент информации находится в своем контексте.

#### **4. Преимущества Данного Архитектурного Подхода**

1.  **Контекстуальная Гибкость:** Информация не заперта в файле. Один и тот же `ContentBlock` может быть частью проекта, а затем, после его завершения, связь может быть изменена на `IS_PART_OF -> Resource`.
2.  **Мощные Запросы:** Появляется возможность задавать сложные вопросы к базе знаний:
    *   "Покажи все задачи, назначенные на Анну Смирнову, по всем активным проектам".
    *   "Какие решения были приняты на встречах, где присутствовал Иван Петров?"
    *   "Найди все ключевые идеи, связанные с ресурсом 'API Design'".
3.  **Отсутствие Дублирования:** Сущности `Person`, `Organization` создаются один раз и переиспользуются во всем графе. Изменение имени "Анна Смирнова" в одном месте обновит его везде.
4.  **Масштабируемость:** Модель легко расширяется новыми типами узлов (например, `Meeting`, `Product`) и отношений без изменения существующей структуры.
5.  **Целостность:** Метаданные проекта (дедлайн, статус) хранятся в одном месте — на узле `Project`, что обеспечивает целостность данных.
