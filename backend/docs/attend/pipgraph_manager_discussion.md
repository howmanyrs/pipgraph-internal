### Анализ процесса в `add_episode`

Функция `add_episode` — это оркестратор, который выполняет последовательность операций. Нас интересует вот эта критически важная часть:

1.  `extracted_nodes = await extract_nodes(...)`
    *   **Что делает:** Берет сырой текст (`episode_body`) и с помощью LLM извлекает из него упоминания сущностей.
    *   **Результат:** Список "сырых" объектов `EntityNode`. На этом этапе Graphiti еще **не знает**, существуют ли эти сущности в графе. "Vladimir Ivanov" здесь — это просто новый объект `EntityNode` с уникальным временным UUID.
    *   **Это наша первая возможная точка интервенции (но не лучшая).**

2.  `nodes, uuid_map, node_duplicates = await resolve_extracted_nodes(...)`
    *   **Что делает:** Это самая важная для вас функция. Она берет "сырые" узлы из шага 1, ищет в Neo4j похожие существующие узлы (`search`), а затем использует LLM, чтобы решить, является ли "сырой" узел дубликатом существующего.
    *   **Результат:**
        *   `nodes`: Финальный список узлов, которые будут в графе. Если "Vladimir Ivanov" был сопоставлен с существующим узлом, здесь будет *существующий* узел. Если нет — *новый*.
        *   `uuid_map`: Словарь, который сопоставляет временный UUID "сырого" узла с финальным UUID узла в графе. `{'uuid_сырого_Vladimir_Ivanov': 'uuid_нового_Vladimir_Ivanov'}`.
    *   **Это идеальная точка для интервенции.**

3.  `extracted_edges = await extract_edges(...)`
    *   **Что делает:** Определяет связи между *финальными, разрешенными* узлами из шага 2.

4.  `(Далее в коде)`: Сохранение узлов и связей в Neo4j.

### Архитектурное решение: Класс-обертка `PipGraphManager`

Вам не нужно изменять исходный код Graphiti. Вместо этого вы создаете свой класс, который будет использовать `graphiti.Graph` как сервис, вызывая его внутренние функции в нужном вам порядке и вставляя между ними логику взаимодействия с пользователем.

Вот как будет выглядеть этот процесс:

#### Шаг 1: Создание обертки и вызов первого этапа (Извлечение)

#### Шаг 2: Вызов второго этапа (Разрешение) и точка интервенции

Теперь у нас есть `extracted_nodes`. Мы просим Graphiti сопоставить их с существующей базой.

```python
# ... продолжение метода process_note ...

        # 2. Вызываем вторую ключевую функцию
        resolved_nodes, uuid_map, node_duplicates = await resolve_extracted_nodes(
            self.graph.clients,
            extracted_nodes,
            episode,
            # ... другие параметры ...
        )

        # 3. --- ВОТ ОНА, ГЛАВНАЯ ТОЧКА ИНТЕРВЕНЦИИ ---
        # Анализируем результаты разрешения
        final_nodes = []
        for original_node in extracted_nodes:
            final_uuid = uuid_map.get(original_node.uuid)
            
            # Находим соответствующий ему финальный узел
            final_node = next((n for n in resolved_nodes if n.uuid == final_uuid), None)
            
            # Условие: если временный UUID узла совпадает с финальным,
            # значит, resolve_extracted_nodes НЕ НАШЕЛ дубликат и решил создать НОВЫЙ узел.
            if original_node.uuid == final_node.uuid:
                # Это новая сущность! Спрашиваем пользователя.
                print(f"Найдена новая сущность: '{final_node.name}'.")
                description = input("Кто это или что это? Введите короткое описание: ")
                
                # Обогащаем узел информацией от пользователя
                if description:
                    final_node.attributes['user_description'] = description
                    final_node.summary = f"{final_node.summary}. User note: {description}".strip()
            
            final_nodes.append(final_node)
```

### Решение второй задачи (заметка без автора)

Этот же подход идеально решает и вторую задачу.

*   **Где перехватывать:** После шага 2 (`resolve_extracted_nodes`).
*   **Логика:**
    1.  Проанализировать `extracted_nodes`. Есть ли среди них `Note` или что-то похожее?
    2.  Проанализировать будущие связи (после вызова `extract_edges`). Связана ли эта `Note` с какой-либо `Person` или `Project`?
    3.  **Если нет**, запускается ваша логика:
        > **Система:** "Я вижу заметку, но не понимаю, к чему она относится. Связать ее с существующей сущностью? (начните вводить имя)"
    4.  Пользователь вводит "Проект ИИ". Вы делаете поиск по `graph.store`, находите узел "Проект ИИ".
    5.  Вы **вручную добавляете** в список `extracted_edges` новую связь между узлом `Note` и узлом "Проект ИИ".
    6.  Только после этого вызываете финальное сохранение.

### Итог и преимущества такого подхода

1.  **Не нужно менять библиотеку:** Вы не трогаете `graphiti_core`. Все обновления библиотеки не сломают ваш код.
2.  **Полный контроль:** Вы полностью контролируете процесс, вставляя свою логику в критически важных точках между вызовами модулей Graphiti.
3.  **Модульность:** Graphiti используется как набор инструментов (toolkit), а не как "черный ящик". Вы берете нужные вам инструменты (`extract_nodes`, `resolve_extracted_nodes`) и строите из них свой уникальный процесс.
