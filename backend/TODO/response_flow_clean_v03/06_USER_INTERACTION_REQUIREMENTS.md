# Требования к User Interaction и Логике Статусов

**Дата создания:** 2025-11-18
**Статус:** Requirements Definition
**Контекст:** Obsidian Plugin <-> PipGraph Backend

---

## 1. Философия: "No Dead Ends" (Никаких тупиков)

Система не должна задавать вопросы, на которые можно ответить просто "Нет".
Каждое взаимодействие должно вести к **уточнению** структуры графа.

*   ❌ **Плохо:** "Это проект?" -> [Да] / [Нет]
    *   *Если "Нет", система в тупике. Заметка осталась сиротой.*
*   ✅ **Хорошо:** "Это похоже на **Новый Проект 'X'**. Или это часть существующего?" -> [Создать 'X'] / [Выбрать другое] / [Это Область]

### Принцип Проактивности
Пока статус не `confirmed`, система держит гипотезу в состоянии `pending`. Она не просто ждет, она **предлагает лучшее, что смогла найти**, и дает удобные кнопки для быстрой коррекции.

## 2. Сценарии взаимодействия (Obsidian UI)

### L1/L2: Контекст Заметки (Linking)

Здесь мы решаем судьбу файла. Вариант "Reject" невозможен, так как заметка должна где-то жить.

**Сценарий А: Высокая уверенность (New Project)**
*   **Ситуация:** LLM уверена на 90%, что это новый проект.
*   **UI Сообщение:** 
    > 🚀 **Новый Проект?**
    > Похоже, вы начинаете "Редизайн Сайта".
*   **Кнопки:**
    1.  `[Подтвердить]` (Создать Project и связь)
    2.  `[Это 'Area']` (Сменить тип на Area)
    3.  `[Выбрать другое...]` (Открыть поиск по существующим контейнерам)

**Сценарий Б: Неопределенность (Ambiguity)**
*   **Ситуация:** Текст похож и на проект "Маркетинг", и на проект "Продажи".
*   **UI Сообщение:** 
    > 🔗 **Связи**
    > К чему относится эта заметка?
*   **Кнопки:**
    1.  `[Проект 'Маркетинг']` (Confidence 60%)
    2.  `[Проект 'Продажи']` (Confidence 35%)
    3.  `[Найти...]` (Ручной поиск)

**Сценарий В: Коррекция (Constructive Correction)**
*   **Пользователь нажимает:** `[Выбрать другое...]`
*   **Действие:** Пользователь выбирает из списка "Archive".
*   **Результат:** Система создает связь с `Archive`, и статус становится `resolved_as_archive`.

### L3: Смысловые Сущности (Extraction)

Здесь "Reject" допустим только в смысле "Это шум/неважно". Но лучше уточнять тип.

**Сценарий: Обнаружена Задача**
*   **Ситуация:** В тексте "Надо бы купить молока".
*   **UI Сообщение:**
    > ✅ **Найдена задача**
    > "Купить молоко"
*   **Кнопки:**
    1.  `[Добавить]` (Создать Entity Task)
    2.  `[Редактировать]` (Изменить формулировку)
    3.  `[Это не задача]` -> *Меню:* `[Это просто текст / Шум]`, `[Это Идея]`.

## 3. Обновленная модель UserCheckStatus

Нода `UserCheckStatus` должна хранить не просто бинарный выбор, а **семантику принятого решения**.

### Поля ноды (Graph Schema)

```cypher
(:UserCheckStatus {
    // Идентификация
    id: "check_123",
    timestamp: datetime(),
    
    // Состояние
    status: "resolved",  // Вместо confirmed/rejected
    
    // Детали решения (Outcome)
    outcome_type: "linked_existing", // linked_new, reclassified, ignored
    
    // Что предлагала система (для аналитики)
    system_proposal_type: "Project",
    system_proposal_name: "Редизайн",
    
    // Что выбрал пользователь
    final_selection_type: "Area",
    final_selection_name: "Дизайн (Общее)",
    
    // Комментарий (если пользователь ввел)
    user_feedback: "Это не проект, это общая папка"
})
```

### Типы Outcomes (Исходов)

1.  **`confirmed_proposal`**: Пользователь согласился с предложением системы.
2.  **`linked_alternative`**: Пользователь выбрал другой контейнер из списка предложенных.
3.  **`created_custom`**: Пользователь создал совершенно новый контейнер вручную.
4.  **`reclassified`**: Пользователь сменил тип (например, Project -> Area).
5.  **`dismissed_as_noise`**: (Только для L3) Пользователь решил, что сущность не нужна.

## 4. Требования к Плагину (Obsidian Client)

Для реализации такой логики плагин должен быть "умным":

1.  **Actionable Notifications:** Уведомление — это не просто текст, это мини-форма с кнопками действий.
2.  **Search Interface:** При нажатии "Выбрать другое" должно открываться модальное окно с быстрым поиском по существующим Projects/Areas в Neo4j (через API бэкенда).
3.  **Context Preservation:** Если пользователь игнорирует уведомление, оно должно оставаться в "Центре уведомлений" плагина или висеть иконкой рядом с заголовком заметки, пока не будет решено.

## 5. Workflow Implications

Это меняет логику `process_response_node` в LangGraph:

*   Вместо `if response == 'yes'`, мы анализируем объект `decision`.
*   **Пример ответа от клиента:**
    ```json
    {
      "check_id": "...",
      "decision": "link_existing",
      "target_id": "uuid_of_area_design" 
    }
    ```
*   Бэкенд должен выполнить `MERGE` связи с указанным `target_id`, а не с тем, что он сам придумал изначально.
