# Implementation Roadmap V3: Constructive Interaction

**Дата обновления:** 2025-11-18
**Статус:** Active Plan
**Версия:** 3.0
**Стратегия:** Top-Down -> Context Injection -> Actionable UI

---

## Обзор плана

Мы строим систему, которая не просто "спрашивает разрешения", а **активно помогает** структурировать знания. Главная задача — реализовать цикл: `Предложение Системы -> Коррекция Пользователя -> Фиксация в Графе`.

### Этапы реализации

1.  **Phase 1: PARA Identification Engine.** Научить систему находить правильное место для заметки (Project/Area).
2.  **Phase 2: Smart Linking Logic.** Реализовать обработку сложных решений пользователя (Link Existing / Create New).
3.  **Phase 3: Context Extraction.** Подключить Graphiti с инъекцией контекста.
4.  **Phase 4: Frontend Loop.** Замкнуть UI на бэкенд.

---

## Phase 1: PARA Identification Engine (Дни 1-2)

**Цель:** Система должна уметь анализировать текст и предлагать **конкретный** существующий контейнер или создание нового.

*   **Задача 1.1:** Реализовать `PipGraphManager.identify_best_para_match(text)`.
    *   Шаг А: Сделать embedding-поиск по существующим Project/Area (найти кандидатов).
    *   Шаг Б: Спросить LLM: "Текст похож на эти проекты? Или это что-то новое?".
    *   *Output:* `{proposal_type, proposal_name, confidence, is_new}`.
*   **Задача 1.2:** Написать Unit-тесты для матчинга.
    *   Тест: Текст про "редизайн" должен матчиться с проектом "Website Redesign".
*   **Задача 1.3:** Убрать старый код `classify_note` (простая классификация типов без привязки).

## Phase 2: Smart Linking Logic (Дни 3-4)

**Цель:** Научить бэкенд понимать не только "Да", но и "Нет, это другое".

*   **Задача 2.1:** Реализовать `PipGraphManager.process_linking_decision(decision)`.
    *   Логика: Если `link_existing` -> Найти UUID -> Создать связь.
    *   Логика: Если `create_custom` -> Создать ноду -> Создать связь.
*   **Задача 2.2:** Реализовать `UserCheckStatus` с полями `outcome` и `final_selection`.
    *   Нам нужно знать, *что именно* выбрал пользователь в итоге, а не просто "confirmed".
*   **Задача 2.3:** Создать ноду графа `commit_link_node`.
    *   Это точка фиксации. После неё заметка официально "в системе".

## Phase 3: Context Extraction (Дни 5-6)

**Цель:** Извлекать смыслы, используя плоды трудов Фазы 1 и 2.

*   **Задача 3.1:** Настроить `extract_entities_with_context`.
    *   Промпт должен принимать: `Current Project: {name}`.
    *   Schema: Whitelist (`Task`, `Person`, `Decision`).
*   **Задача 3.2:** Реализовать фильтрацию "шума".
    *   Если LLM вернула "Meeting" (запрещенный тип), мы молча отбрасываем это до того, как спросить пользователя.
*   **Задача 3.3:** Реализовать `finalize_node` (Bulk Save).
    *   Сохраняем только `confirmed` сущности.

## Phase 4: Frontend Loop & Polish (Дни 7-8)

**Цель:** Сделать так, чтобы Obsidian плагин получал данные, с которыми можно работать.

*   **Задача 4.1:** Формат JSON для WebSocket.
    *   Убедиться, что `interrupt` отправляет полный объект `proposal` (с именем, типом и уверенностью), а не просто текст вопроса.
*   **Задача 4.2:** Обработка `user_decision_payload`.
    *   Парсинг ответа от клиента (`target_id`, `new_name` и т.д.) в `NoteWorkflowState`.
*   **Задача 4.3:** E2E Тест "Correction Scenario".
    *   Сценарий: Система предложила "Проект А". Юзер выбрал "Область Б". Система создала связь с "Область Б".

## Definition of Done (Критерии успеха)

1.  ✅ **No Dead Ends:** В коде нет ветки, где ответ "Reject" просто обрывает процесс без результата (заметка должна быть куда-то привязана, хотя бы в `Inbox` или `Archive`).
2.  ✅ **Context is King:** В логах Graphiti видно, что промпт содержит имя проекта, выбранного пользователем.
3.  ✅ **Clean Graph:** В Neo4j нет дублирующихся свойств (`note.project_id` is NULL), все связи на месте.

---

## Риски и Митигация

*   **Риск:** LLM галлюцинирует и предлагает несуществующие проекты.
    *   *Митигация:* В `identify_best_para_match` мы сначала делаем векторный поиск по базе. LLM выбирает *только* из найденных кандидатов или говорит "New".
*   **Риск:** Пользователь долго не отвечает.
    *   *Митигация:* LangGraph сохраняет состояние в SQLite. Процесс висит в памяти (или на диске) бесконечно долго, ожидая `resume`.
