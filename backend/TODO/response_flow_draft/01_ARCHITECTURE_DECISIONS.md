# Архитектурные решения

**Дата обновления:** 2025-11-18
**Статус:** Утверждено для реализации
**Стратегия:** Identification -> Context Injection -> Extraction

---

## Введение

Этот документ фиксирует архитектурные принципы системы обработки заметок. Главное изменение — переход к последовательной обработке, где мы сначала **находим место заметки в графе** (связываем с конкретным Проектом или Областью), и только потом извлекаем из неё смыслы, используя полученный контекст.

---

## Решение 1: Top-Down Workflow (Identification & Linking First)

### Суть решения
Обработка заметки разбивается на две строгие фазы:
1.  **L1/L2 (Identification):** Система анализирует заметку, чтобы найти, к какому **существующему** Проекту или Области в базе знаний она относится. Если подходящего контейнера нет — предлагает создать новый.
2.  **L3 (Content Extraction):** Только после того, как заметка получила "прописку" (связь `[:IS_PART_OF]`), запускается извлечение сущностей.

### Почему это решение
*   **Устранение двусмысленности:** Одна и та же фраза (например, "Заказать дизайн") имеет разный смысл в контексте проекта "Ремонт квартиры" и проекта "Запуск веб-сайта".
*   **Связность графа:** Мы приоритезируем развитие существующей структуры (Projects/Areas), предотвращая создание изолированных островов информации.

---

## Решение 2: Context-Injected Extraction

### Суть решения
При запуске L3 (Extraction) мы иньецируем метаданные найденного контейнера в промпт LLM.

**Пример:**
*   *Без контекста:* LLM видит текст "Купить краску". -> Создает абстрактную задачу.
*   *С контекстом:* Система передает: "Контекст: Проект 'Ремонт Детской'". -> LLM создает задачу "Купить краску для детской", понимая специфику.

### Почему это решение
*   Это решает проблему "плоской" экстракции, когда Graphiti создает слишком общие или оторванные от реальности сущности. Контекст родительского узла (PARA) действует как смысловой фильтр.

---

## Решение 3: No-Cache Policy (Отказ от кэширования)

### Суть решения
Мы отказываемся от дублирования связей в атрибутах нод.
*   **Note Node:** Не хранит `project_id` или `para_type`.
*   **Entity Node:** Не хранит `status` или `check_id`.

**Источник истины — только графовые связи.**

### Реализация
Вместо чтения свойств, мы используем навигацию по графу:
```cypher
// Узнать, к какому проекту относится заметка
MATCH (n:Note {path: $path})-[:IS_PART_OF]->(container)
RETURN container
```

### Почему это решение
*   **Гарантия консистентности:** Исключается ситуация, когда заметка перемещена в другой проект (связь изменилась), а поле `project_id` забыли обновить.
*   **Простота записи:** Код `PipGraphManager` освобождается от логики синхронизации атрибутов.

## Решение 4: Strict Schema & Whitelist

### Суть решения
Вместо того чтобы позволять LLM придумывать любые типы сущностей, мы используем жесткий список разрешенных типов (Whitelist) на этапе экстракции.

### Конфигурация
*   **Разрешено:** `Concept` (ключевые идеи), `Person`, `Task`, `Decision`.
*   **Игнорируется:** Всё остальное (даты, инструменты, форматы файлов и прочий "шум").

### Почему это решение
*   Graphiti по умолчанию склонен к избыточной гранулярности (NER). Ограничение схемы заставляет модель обобщать и выделять только главные смысловые блоки.

---

## Решение 5: Simplified UserCheckStatus

### Суть решения
История решений пользователя хранится в отдельных узлах `UserCheckStatus`.
Для MVP мы упрощаем модель данных: храним только финальный вердикт (`confirmed` / `rejected`) и комментарий.

### Структура
```cypher
(entity:EntityNode)-[:HAS_CHECK {is_current: true}]->(status:UserCheckStatus {val: "confirmed"})
```

### Почему это решение
*   Это минимально необходимый набор данных для реализации `Human-in-the-loop` и последующего дообучения системы (анализ ложноположительных срабатываний).
