# PipGraph — Интеллектуальный Inbox для вашей базы знаний

**PipGraph** — это система GTD (Getting Things Done) для персональной базы знаний (PKM). Она берет на себя самую сложную часть работы с заметками: **организацию и распределение** — чтобы вы могли сосредоточиться на содержании, а не на структуре.

## Проблема: Хаос во «Входящих»

Каждый, кто ведет заметки в Obsidian, Notion или просто в папках с Markdown-файлами, сталкивается с одними и теми же проблемами:

**Куда положить заметку?**
- Папка «Работа» или «Проекты»? А если это и то, и другое?
- Создать новую категорию или использовать существующую?
- Как быть с заметкой, которая относится к нескольким темам?

**Inbox превращается в свалку**
- Новые заметки копятся во «Входящих» днями и неделями
- Разбор завалов требует времени и усилий
- Чем больше база, тем сложнее принять решение о классификации

**Структура распадается**
- Папки множатся хаотично
- Теги теряют смысл
- Связи между заметками не видны

Это не проблема лени или дисциплины. **Это архитектурная проблема**: человеческий мозг плохо справляется с классификацией растущих объемов информации.

## Решение: GTD для заметок

PipGraph применяет принципы GTD к управлению знаниями:

| GTD для задач | PipGraph для заметок |
|---------------|---------------------|
| Capture — записать мысль | Создать заметку в любом формате |
| Clarify — что это значит? | LLM извлекает суть и теги |
| Organize — куда положить? | Система предлагает подходящие папки |
| Reflect — регулярный обзор | Граф связей показывает картину |
| Engage — действовать | Экспорт в чистую структуру |

**Ключевое отличие**: PipGraph не заставляет вас выбирать одну методологию. Вы организуете хранение так, как удобно вам:
- Папки по темам
- PARA (Projects, Areas, Resources, Archives)
- Zettelkasten с плоским списком
- Любая гибридная структура

## Как это работает

### 1. Пользователь создает структуру

Вы создаете папки-узлы для хранения заметок. Каждая папка — это узел в графовой базе данных с описанием:

```
📁 Работа/
   📁 Проект: Редизайн сайта
      description: "Редизайн корпоративного сайта компании.
                   Сроки: Q1 2026. Команда: 3 разработчика."
   📁 Проект: API миграция
      description: "Переход с REST v1 на GraphQL.
                   Включает документацию и тестирование."

📁 Области/
   📁 Программирование/Python
      description: "Заметки по Python: паттерны, библиотеки,
                   best practices, решения проблем."
   📁 Здоровье
      description: "Тренировки, питание, медицинские записи."
```

**Метаданные папок** адаптируются под вашу методологию:
- Тип заметки (проект, область, ресурс, архив)
- Сроки и дедлайны
- Приоритеты
- Любые пользовательские поля

### 2. Заметка попадает в Inbox

Новая заметка привязывается хотя бы к одному узлу (в MVP-версии). Это отправная точка для обработки:

```markdown
# Встреча с заказчиком 15.01

Обсудили требования к новому API:
- Аутентификация через OAuth 2.0
- Rate limiting: 1000 req/min
- Версионирование через заголовки

Заказчик хочет видеть прототип к концу месяца.
```

### 3. LLM анализирует содержание

При обработке заметки система:

**Извлекает сущности и теги:**
- `#api`, `#oauth`, `#встреча`
- Упоминания: "заказчик", "прототип", "rate-limiting"

**Суммаризирует с контекстом папки:**
- Информация из заметки добавляется к описанию родительского узла
- Поле `summary` папки становится кратким конспектом всех её заметок
- При добавлении новых заметок summary обновляется

**Теги становятся глобальными:**
- Извлеченные темы связывают заметки независимо от папок
- Заметка в "Работа/API миграция" и заметка в "Обучение/Backend" могут иметь общий тег `#api`

### 4. Система предлагает распределение

Когда вы добавляете новую заметку в Inbox, PipGraph ищет похожее содержание:

**Гибридный поиск:**
- BM25 (полнотекстовый) — находит точные совпадения терминов
- Векторное сходство — понимает семантическую близость
- Результаты переранжируются для оптимальной релевантности

**Пользователь видит рекомендации:**
```
📥 Новая заметка: "OAuth 2.0 flow для мобильных приложений"

Рекомендуемые папки:
├─ 🎯 92% — Работа/API миграция
│     (Упоминается OAuth в контексте текущего проекта)
├─ 📚 78% — Области/Программирование/Python
│     (Есть заметки по аутентификации)
└─ 📁 65% — Ресурсы/Безопасность
      (Общая тема авторизации)
```

**Подтверждение или автоматизация:**
- Вы выбираете подходящую папку
- Или включаете автоматическое назначение, если система стабильно угадывает

### 5. Экспорт в чистую структуру

Организованные заметки экспортируются как обычные папки с файлами:
- Markdown-файлы с размеченными ссылками `[[связь]]`
- Теги в YAML frontmatter
- Готово для использования в Obsidian или любом редакторе

```markdown
---
tags: [api, oauth, встреча]
links:
  - "[[Проект - API миграция]]"
  - "[[OAuth 2.0 спецификация]]"
created: 2026-01-15
---

# Встреча с заказчиком 15.01
...
```

## Почему это работает

### Проблемы пользователей Obsidian (из анализа сообщества)

| Проблема | Как решает PipGraph |
|----------|---------------------|
| **«Куда положить заметку, если она относится к двум папкам?»** | Заметка может быть связана с несколькими узлами. Физическое расположение — в одном месте, связи — везде. |
| **«Папки vs ссылки — что выбрать?»** | Обе модели работают. Папки дают структуру, граф показывает связи. |
| **«База растет, поддерживать порядок становится "второй работой"»** | LLM берет на себя классификацию. Вы только подтверждаете. |
| **«Теги множатся и теряют смысл»** | Система извлекает теги автоматически и унифицирует их. |
| **«Как организовать Inbox, чтобы он не превращался в свалку?»** | Это ядро PipGraph — интеллектуальный Inbox с рекомендациями. |

### Архитектурные принципы

1. **Неразрушающая обработка**
   - Тело заметки остается неизменным
   - Метаданные пишутся только в YAML frontmatter
   - Оригинал всегда сохраняется

2. **Human-in-the-Loop**
   - Система предлагает, человек решает
   - Автоматизация включается только по желанию пользователя
   - Всегда можно вмешаться и переопределить

3. **Граф как основа**
   - Информация хранится в Neo4j как узлы и связи
   - Папки, заметки, теги, сущности — всё связано
   - Сложные запросы через обход графа

4. **Гибкая методология**
   - Никаких жестких схем
   - Поддержка PARA, Zettelkasten, папочной иерархии
   - Структура адаптируется под пользователя

### Модель данных

**Episodic (Заметка)**
```
(:Episodic {
  uuid,
  name,           // путь к файлу
  content,        // текст заметки
  created_at,
  valid_at        // дата события в заметке
})
```

**Entity (Узел PARA)**
```
(:Entity:Project|:Area|:Resource|:Archive {
  uuid,
  name,              // название папки/категории
  summary,           // автоматический конспект содержимого
  name_embedding,    // вектор для семантического поиска
  attributes,        // пользовательские метаданные
  created_at
})
```

**Связи**
- `(:Episodic)-[:MENTIONS]->(:Entity)` — заметка относится к категории
- `(:Entity)-[:RELATES_TO]->(:Entity)` — связь между категориями

## Быстрый старт

### Требования
- Python 3.12+
- Node.js 18+
- Neo4j 5.x
- API-ключ OpenRouter (для LLM)

### Backend

```bash
cd backend/
uv venv && source .venv/bin/activate
uv pip install -r requirements.txt

# Настройка переменных окружения
cp .env.example .env
# Отредактируйте: OPENROUTER_API_KEY, NEO4J_URI, NEO4J_USER, NEO4J_PASSWORD

# Запуск сервера
uvicorn app.api.main:app --reload
# API: http://localhost:8000
# Документация: http://localhost:8000/docs
```

### Web UI

```bash
cd pipgraph-web/
npm install
npm run dev
# Интерфейс: http://localhost:3000
```

## Философия проекта

PipGraph не пытается заменить ваш редактор заметок. Obsidian, Notion, VS Code — используйте что удобно.

PipGraph решает одну конкретную проблему: **куда положить эту заметку?**

Это интеллектуальный Inbox, который:
- Понимает содержание ваших заметок
- Знает структуру вашей базы знаний
- Предлагает оптимальное место
- Позволяет вам принять финальное решение

*Порядок в заметках — без боли.*
